volcano核心概念入门

Volcano 核心概念入门学习讲义

一、Volcano 的定位与解决的问题（1小时）

1.1 Kubernetes 原生调度器的局限

  • 原子性调度缺失：无法保证批量作业的所有 Pod 同时启动
  • 资源碎片问题：无法全局优化资源分配，导致资源浪费
  • 作业级调度缺失：缺乏作业级别的优先级、公平调度和队列管理
  • 高级调度策略不足：缺乏拓扑调度、任务依赖等批处理场景特性

1.2 Volcano 的核心价值

  • 批处理场景优化：专为 AI/大数据/HPC 工作负载设计
  • 增强调度能力：
    • Gang Scheduling（All-or-Nothing）
    • 公平调度（DRF）
    • 队列管理
    • 任务依赖（DAG）
    • 拓扑感知调度
  • Kubernetes 原生集成：通过 CRD 扩展 Kubernetes API

1.3 典型应用场景

场景 问题 Volcano 解决方案
TensorFlow 分布式训练 需要所有 Worker 同时启动 Gang Scheduling
Spark 批处理作业 资源碎片导致部分 Executor 无法启动 Binpack 资源装箱
多租户环境 团队间资源争抢 队列配额管理
MPI 高性能计算 跨节点网络拓扑要求 拓扑感知调度
Kubeflow 流水线 任务间依赖关系 DAG 工作流支持

  • 实现 Gang Scheduling 的基础抽象

  • minMember：确保指定数量的 Pod 同时就绪
  • 状态机：

2.2 VolcanoJob：扩展的作业定义

概念：

  • 扩展 Kubernetes 原生 Job
  • 支持多任务模板（如 Master/Worker 角色）

核心结构：

任务类型：

  1. 主任务（如 Chief）：通常只有 1 个副本
  2. 工作节点（Worker）：多个副本并行计算
  3. 参数服务器（PS）：存储模型参数（可选）
  4. 辅助任务：数据预处理、日志收集等

2.3 Queue：资源队列与多租户管理

概念：

  • 资源分配的逻辑容器
  • 实现多租户资源隔离和优先级控制

核心配置：

队列特性：

  • 权重分配：DRF（主导资源公平）算法分配资源
  • 资源回收：低优先级作业可被抢占
  • 状态管理：
    • Open：接受新作业
    • Closed：停止接受新作业
    • Draining：排空现有作业

2.4 核心概念关系

职责分离（Separation of Concerns）

对象 职责 Queue的操作目标
Volcano Job 描述业务逻辑（如Task/Pod定义） 不直接参与资源调度
PodGroup 抽象调度需求（如minAvailable） 资源分配的计算单元
Queue 管理资源配额（如权重/容量） 以PodGroup为单位分配资源

关键路径：

  1. 用户定义 Job 的 minAvailable 和 queue →
  2. Volcano 自动创建同名的 PodGroup →
  3. 调度器基于 PodGroup 信息 + Queue 资源约束做原子调度 →
  4. 组内所有 Pod 统一绑定节点。

调度流程中的关联关系

  • Queue的关联对象：

    • 直接关联：PodGroup（通过.spec.queue字段绑定）
    • 间接关联：Volcano Job（通过PodGroup桥接）

  • 关键结论：

    ✅ Queue内部只感知PodGroup，不会直接处理Volcano Job对象。
    ❌ Volcano Job从不进入Queue，它只是PodGroup的创建者。

为什么不直接调度job，而是使用一个新的CRD podgroup

解耦调度单元与业务单元

  • Job 的局限性：

    • 本质是业务描述（如 Task 配置、镜像、命令），包含过多与调度无关的字段（如容器环境变量）。
    • 若直接调度，调度器需过滤大量无关数据 → 性能下降。

  • PodGroup 的优势：

    • 轻量化调度上下文：仅保留调度关键字段：

    • 调度器只需关注 PodGroup 而非整个 Job → 决策效率提升。

三、调度核心原理解析（1小时）

3.1 Gang Scheduling 工作流程

  1. 作业提交：创建 VolcanoJob 和关联的 PodGroup
  2. 资源检查：调度器验证集群是否满足 minAvailable 要求
  3. 资源预留：锁定所需资源（防止被其他作业占用）
  4. 原子绑定：同时绑定所有 Pod 到节点
  5. 状态更新：
    • 成功：PodGroup 状态 → Running
    • 失败：PodGroup 状态 → Pending（持续重试）

3.2 调度关键决策点

  1. 队列选择：根据权重分配队列资源配额
  2. 作业排序：基于优先级/提交时间排序待调度作业
  3. 节点过滤：排除不满足要求的节点
  4. 任务优先级：同一作业内任务排序（如先调度 Chief）
  5. 资源分配：使用 Binpack/Spread 算法分配节点

3.3 调度器插件体系

Volcano 通过插件实现调度策略：

  • Enqueue：作业入队决策
  • Allocate：资源分配决策
  • Preempt：抢占策略
  • Reclaim：资源回收
  • Overcommit：超卖策略
