volcano调度器Enqueue-Action详解

Volcano 调度器 Enqueue Action 源码级详解

Enqueue Action 是 Volcano 调度工作流中的第一个关键步骤，负责将符合条件的待调度作业(Job)加入调度队列（标记为 Inqueue 状态）。以下结合源码（v1.8.0）详细解析其实现机制：

一、Enqueue Action 入口函数

源码位置：pkg/scheduler/actions/enqueue/enqueue.go

二、核心处理流程详解

1. 作业筛选与分组

  • 从 Session 缓存中筛选所有状态为 Pending 的作业
  • JobInfo 结构体（pkg/scheduler/api/job_info.go）封装了作业的完整信息

2. 队列排序机制

  • 使用 QueueOrderFn 对队列排序（默认为 DRF 算法）
  • 自定义排序可通过 proportion 插件实现（pkg/scheduler/plugins/proportion）

3. 作业入队检查

  • 通过插件链实现多条件检查
  • 关键检查插件：
    • Gang 插件：检查 minAvailable 资源是否满足
    • ResourceQuota 插件：检查队列配额
    • SLA 插件：检查作业等待时间

4. Gang 插件资源检查

源码位置：pkg/scheduler/plugins/gang/gang.go

5. ResourceQuota 插件队列配额检查

源码位置：pkg/scheduler/plugins/proportion/proportion.go

6. SLA 插件超时处理

源码位置：pkg/scheduler/plugins/sla/sla.go

三、状态更新机制

1. 作业状态更新

  • 将 PodGroup 状态从 Pending 更新为 Inqueue
  • 更新后的状态会同步到 Kubernetes API Server

2. 队列状态更新

  • 更新队列的作业计数
  • 资源配额更新由 proportion 插件在后续处理

四、Enqueue Action 的插件扩展点

扩展点 作用 默认实现
QueueOrderFn 定义队列排序逻辑 proportion 插件
JobOrderFn 定义作业在队列内的排序逻辑 FIFO
JobEnqueueableFn 定义作业入队条件检查 gang/resoucequota
JobReadyFn 定义作业是否准备好调度 gang 插件

五、Enqueue 流程时序图

sequenceDiagram
    participant Scheduler
    proportion->>Scheduler: 提供QueueOrderFn
    gang->>Scheduler: 提供JobEnqueueableFn
    sla->>Scheduler: 提供JobEnqueueableFn
    
    Scheduler->>Cache: 获取Pending作业
    Cache-->>Scheduler: 返回Job列表
    
    loop 每个队列
        Scheduler->>proportion: 调用QueueOrderFn排序队列
        proportion-->>Scheduler: 返回队列排序结果
        
        loop 队列内作业
            Scheduler->>gang: 调用JobEnqueueableFn
            gang-->>Scheduler: 返回资源检查结果
            Scheduler->>sla: 调用JobEnqueueableFn
            sla-->>Scheduler: 返回超时检查结果
            
            alt 检查通过
                Scheduler->>API Server: 更新PodGroup状态(Inqueue)
                Scheduler->>Cache: 更新队列状态
            else 检查失败
                Scheduler->>Log: 记录拒绝原因
            end
        end
    end

六、设计要点分析

  1. 插件化架构：

    • 所有核心逻辑通过插件接口实现
    • 支持自定义扩展而不修改核心代码

  2. 两级排序机制：

    • 队列级排序：决定哪个队列优先调度
    • 作业级排序：决定队列内作业的调度顺序

  3. 条件检查链：

    • 多个检查插件形成”与”逻辑关系
    • 任一插件拒绝则作业不入队

  4. 状态原子更新：

    • 作业状态更新与队列计数更新同步进行
    • 通过 Session 保证操作原子性

  5. 资源预留设计：

    • 入队时不立即占用资源
    • 实际资源分配在 Allocate Action 进行

Enqueue Action 通过这种设计实现了高效的作业准入控制，为后续的资源分配阶段提供了合格的候选作业集合，同时保证了集群资源的合理利用和作业的公平调度。
