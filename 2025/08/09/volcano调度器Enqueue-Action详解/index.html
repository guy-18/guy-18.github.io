volcano调度器Enqueue-Action详解

Volcano 调度器 Enqueue Action 源码级详解

Enqueue Action 是 Volcano 调度工作流中的第一个关键步骤，负责将符合条件的待调度作业(Job)加入调度队列（标记为 Inqueue 状态）。以下结合源码（v1.8.0）详细解析其实现机制：

一、Enqueue Action 入口函数

源码位置：pkg/scheduler/actions/enqueue/enqueue.go

二、核心处理流程详解

1. 作业筛选与分组

  • 从 Session 缓存中筛选所有状态为 Pending 的作业
  • JobInfo 结构体（pkg/scheduler/api/job_info.go）封装了作业的完整信息

2. 队列排序机制

  • 使用 QueueOrderFn 对队列排序（默认为 DRF 算法）
  • 自定义排序可通过 proportion 插件实现（pkg/scheduler/plugins/proportion）

3. 作业入队检查

  • 通过插件链实现多条件检查
  • 关键检查插件：
    • Gang 插件：检查 minAvailable 资源是否满足
    • ResourceQuota 插件：检查队列配额
    • SLA 插件：检查作业等待时间

4. Gang 插件资源检查

源码位置：pkg/scheduler/plugins/gang/gang.go

5. ResourceQuota 插件队列配额检查

源码位置：pkg/scheduler/plugins/proportion/proportion.go

6. SLA 插件超时处理

源码位置：pkg/scheduler/plugins/sla/sla.go

三、状态更新机制

1. 作业状态更新

  • 将 PodGroup 状态从 Pending 更新为 Inqueue
  • 更新后的状态会同步到 Kubernetes API Server

2. 队列状态更新

  • 更新队列的作业计数
  • 资源配额更新由 proportion 插件在后续处理

四、Enqueue Action 的插件扩展点

扩展点 作用 默认实现
QueueOrderFn 定义队列排序逻辑 proportion 插件
JobOrderFn 定义作业在队列内的排序逻辑 FIFO
JobEnqueueableFn 定义作业入队条件检查 gang/resoucequota
JobReadyFn 定义作业是否准备好调度 gang 插件

六、设计要点分析

  1. 插件化架构：

    • 所有核心逻辑通过插件接口实现
    • 支持自定义扩展而不修改核心代码

  2. 两级排序机制：

    • 队列级排序：决定哪个队列优先调度
    • 作业级排序：决定队列内作业的调度顺序

  3. 条件检查链：

    • 多个检查插件形成”与”逻辑关系
    • 任一插件拒绝则作业不入队

  4. 状态原子更新：

    • 作业状态更新与队列计数更新同步进行
    • 通过 Session 保证操作原子性

  5. 资源预留设计：

    • 入队时不立即占用资源
    • 实际资源分配在 Allocate Action 进行

Enqueue Action 通过这种设计实现了高效的作业准入控制，为后续的资源分配阶段提供了合格的候选作业集合，同时保证了集群资源的合理利用和作业的公平调度。

讲解二

结合最新版本的 Volcano 源码，我来为你详细解析 Enqueue Action 的具体流程。Enqueue Action 是 Volcano 调度器工作流中的第一个关键步骤，它决定了哪些作业（PodGroup）有资格进入调度队列，后续才能被分配资源。

🧠 核心概念与作用

在深入了解代码之前，我们先明确 Enqueue Action 的核心使命：

  • “守门员”：它对所有处于 Pending 状态的 PodGroup（在调度器内部封装为 JobInfo）进行筛选。只有当集群当前可用的空闲资源能够满足一个 PodGroup 所声明的最小资源需求（即 minAvailable，通常代表该作业成功运行所需的最少 Pod 数量）时，这个 PodGroup 才会被允许进入队列（状态变为 Inqueue）。
  • 避免资源死锁与碎片化：这种“All or Nothing”的预检机制至关重要。它防止了因资源不足而仅调度部分 Pod，导致分布式任务无法完成的同时又占着资源不放的情况（资源死锁），同时也减少了大量无法调度的 Pending Pod 对 Kubernetes API Server 的压力和集群资源的碎片化。
  • 状态转换与 Pod 创建触发器：只有 PodGroup 状态变为 Inqueue，volcano-controller-manager 中的 job controller 才会为此 PodGroup 创建其关联的所有 Pod。这些 Pod 随后才会被调度器缓存并进入后续的调度环节（如 Allocate, Backfill 等）。

🔍 Enqueue Action 详细工作流程

以下是 Enqueue Action 在一个调度周期（Session）中的详细工作步骤，结合了最新版本（如 v1.12.0）的源码逻辑：

  1. 调度周期开始：
    Volcano Scheduler 的主循环 scheduler.Run() 会周期性地（由 schedulePeriod 定义，默认约 1 秒）调用 scheduler.runOnce() 或类似逻辑，从而开启一个新的调度会话 (Session)。

  2. 收集待调度的 JobInfo：
    在每个调度周期的开始，调度器会从它的缓存 (SchedulerCache) 中获取所有未被调度的 JobInfo（即其对应的 PodGroup 状态不是 Inqueue 或 Running）。这些 JobInfo 封装了 PodGroup 的信息以及其下的所有 Pod（TaskInfo）。

  3. 执行 Enqueue Action：
    调度器会根据配置（通常在 volcano-scheduler-configmap 中定义）依次执行一系列 Action。Enqueue 通常是第一个被执行的 Action。

  4. 核心检查：jobReady 与 Gang 插件：
    jobReady 函数（或类似逻辑）是决定性的环节。它通常会调用 Gang 插件 的 OnSessionOpen 或其他相关方法。Gang 插件会：

    • 获取该 JobInfo 的 minAvailable 值（源自 PodGroup 的 spec.minAvailable）。
    • 检查集群当前的空闲资源（ss.Nodes 中各个节点未分配的资源之和，可能还会考虑 Overcommitment 和队列权重weight计算后的应得资源deserved）是否能够同时满足 minAvailable 个 Pod 的资源请求（request）。
    • 这个检查是“模拟性”的，并不实际分配资源，旨在快速判断可行性。
    • 如果空闲资源 >= minAvailable 个 Pod 的请求资源之和，则返回 true，否则返回 false。

  5. 状态更新与提交：

    • 对于通过检查的 Job，其状态会在 Session 中被更新为 Inqueue。
    • 在 Session 结束 (CloseSession) 时，所有通过 Enqueue Action 的决策（即 PodGroup 状态变更）会通过 API Server 提交并持久化到 etcd 中。

  6. Controller 触发 Pod 创建：

    • volcano-controller-manager 一直监视着 PodGroup 的状态变化。当它观察到某个 PodGroup 的状态从 Pending 变为 Inqueue 后，会开始创建这个 PodGroup 所关联的所有 Pod（依据对应的 Volcano Job 中 tasks 模板定义）。
    • 这些新创建的 Pod 会被 Volcano Scheduler 的缓存捕获，并在下一个调度周期中成为待调度的对象，进入后续的 Allocate, Backfill 等 Action。

🔧 关键插件与协同

Enqueue Action 严重依赖其他插件来完成其决策：

  • Gang 插件：这是 Enqueue 的核心插件。它强制执行“All or Nothing”的调度策略，确保 minAvailable 得到满足。
  • Priority 插件：用于对多个处于 Pending 状态的 Job 进行排序，决定 Enqueue 检查的先后顺序。
  • Proportion 插件：负责计算和管理队列（Queue） 的资源权重（weight）、保障（guarantee）和上限（capability）。Enqueue Action 在检查集群可用资源时，可能会考虑队列的应得资源（deserved），而不仅仅是节点的全部空闲资源。需要注意的是，队列的 guarantee 和 capability 在 admission 阶段可能缺乏强验证，其最终效果依赖于 Proportion 插件的计算，实际使用中需注意。

⚠️ 重要注意事项

  1. 与 Preempt/Reclaim 的冲突：如果调度配置中同时启用了 enqueue 和 preempt/reclaim，可能会产生逻辑冲突。因为 Enqueue 在资源不足时会阻止 PodGroup 入队，进而导致没有 Pod 被创建。而没有 Pending 状态的 Pod，preempt 和 reclaim action 就找不到抢占或回收的对象，从而无法生效。配置调度策略时需要仔细考虑这一点。
  2. Queue 资源的考量：Enqueue 的决策会受到 PodGroup 所属队列的资源配额影响（通过 Proportion 插件）。但正如前文所述，队列配额的最终效果需要结合 Proportion 插件的具体实现和配置来评估。
