volcano调度器Reclaim-Action详解

通过分析 Volcano 的源码（主要参考 master 分支和最新稳定版，如 v1.9.0），我来为你详细解析 Reclaim Action 的具体过程。Reclaim Action 是 Volcano 调度器中负责跨队列资源回收的核心机制，它确保资源能够按照队列的权重和配额进行公平分配。

🔧 Reclaim Action 的核心作用与设计初衷

核心作用：Reclaim Action 的主要目标是当某个队列（Queue） 的资源使用量超过其应得份额（deserved） 时，从其他可回收（reclaimable） 的队列中回收资源，以满足资源短缺队列的需求。其设计初衷是实现多租户场景下跨队列的公平调度和资源隔离。

与 Preempt Action 的区别：

  • Reclaim: 处理不同队列之间的资源回收。
  • Preempt: 处理同一队列内部（Job 之间或 Task 之间）基于优先级的抢占。

📦 关键概念与前置知识

理解 Reclaim Action 需要先了解几个关键概念：

  • Session: 调度会话的上下文，包含了当前所有待调度的 Job(Jobs)、节点信息(Nodes)、已注册的插件(Plugins)等。
  • Queue: Volcano 的自定义资源，用于对作业进行分组和资源分配。每个 Queue 可以配置 deserved（应得资源量）、capability（资源上限）和 guarantee（保障资源量）。
  • deserved: 队列根据其权重等因素计算出的应得资源量。
  • reclaimable: 队列的一个属性（spec.reclaimable），标记该队列中的资源是否可以被其他队列回收。只有将 reclaimable 设置为 true 的队列，其资源才可能被回收。
  • 层级队列（Hierarchical Queue）: Volcano 最新版本支持队列的层级结构，子队列的资源配额受父队列限制，Reclaim Action 也需要考虑这种层级关系。

🔍 Reclaim Action 详细工作流程

Reclaim Action 的入口点通常在 pkg/scheduler/actions/reclaim/reclaim.go 文件中的 Execute 函数。其工作流程可以概括为以下核心步骤：

  1. 识别资源短缺的队列：
    Reclaim Action 会遍历所有队列，计算每个队列的 deserved 资源量和当前实际已分配（allocated）的资源量。如果某个队列的 allocated 资源小于其 deserved 资源，说明该队列可能资源短缺，有资格发起资源回收。这些队列被称为 “ reclaimor”（回收方）。

  2. 识别资源超用的队列（候选受害者队列）：
    同样，遍历所有队列，寻找那些 allocated 资源超过其 deserved 资源的队列。这些队列是潜在的 “victim”（受害者）。只有那些 reclaimable 字段为 true 的队列才会被考虑作为受害者。

  3. 选择要回收的任务（Pod）：
    对于每一个资源超用的受害者队列，Reclaim Action 会遍历其内部已运行的 Job 和 Pod（TaskInfo），并根据配置的插件（如 Priority 插件）选择最适合被回收的 Pod。通常，优先级较低或创建时间较早的 Pod 会被优先选择。

  4. 模拟回收与安全检查：
    在真正回收之前，会进行模拟和安全检查，这个过程严重依赖 Gang 插件：

    • Gang 插件会检查：如果驱逐（回收）了选中的 Pod，是否会导致其所属的 Job 不满足 minAvailable 约束（即破坏其 Gang 调度策略）。
    • 只有那些被回收后不会破坏原 Job 完整性的 Pod，才会被确定为真正的回收目标。

  5. 执行资源回收：
    一旦确定了要回收的 Pod，调度器会真正地驱逐这些 Pod。这是通过调用 Kubernetes API 删除这些 Pod 来实现的。这些被回收的 Pod 会进入 Terminating 状态，其占用的资源将被释放。

  6. 资源释放与再分配：
    受害者 Pod 被删除后，它们之前占用的资源就被释放了。这些释放出来的资源并不会立即分配给发起回收的队列，而是等待下一个调度周期，由 Enqueue 和 Allocate 等 Action 来重新分配这些资源。此时，由于之前资源短缺的队列有资源需求，它就有机会在后续的调度中获得这些资源。

⚙️ 依赖的插件与配置

Reclaim Action 的具体行为由一系列插件决定：

  • Gang 插件: 进行安全检查的核心插件，确保回收操作不会破坏分布式任务的原子性（”All or Nothing”）。
  • Priority 插件: 用于在受害者队列中选择要回收的 Pod 时，比较 Pod 或 Job 的优先级。
  • Proportion 插件: 负责计算和管理队列的 deserved、capability 等资源配额。
  • Capacity 插件 (用于层级队列): 当启用层级队列功能时（enableHierarchy: true），该插件负责处理跨层级队列的资源分配和回收逻辑。

🧪 结合示例说明

假设有两个队列：

  • queue-online (高优先级): deserved 为 20 核 CPU，reclaimable: false。
  • queue-offline (低优先级): deserved 为 20 核 CPU，reclaimable: true。

当前 queue-offline 中有一个 Job 正在运行，占用了 40 核 CPU（即 allocated > deserved）。此时，如果有一个新的 Job 被提交到 queue-online，需要 10 核 CPU，但集群当前没有足够空闲资源。

  1. Reclaim Action 触发：

    • 识别到 queue-online 的 allocated (0核) < deserved (20核)，资源短缺，成为回收方。
    • 识别到 queue-offline 的 allocated (40核) > deserved (20核)，且 reclaimable: true，成为受害者队列。

  2. 选择与检查：

    • 从 queue-offline 中选择一个或多个合适的 Pod（例如优先级最低的），假设这些 Pod 总共占用 10 核 CPU。
    • Gang 插件检查回收这些 Pod 是否会影响其所属 Job 的 minAvailable。假设检查通过。

  3. 执行回收：

    • 驱逐选中的 Pod，释放 10 核 CPU。

  4. 资源再分配：

    • 在下一个调度周期，queue-online 中的新 Job 有机会通过 Allocate Action 分配到这 10 核 CPU，从而成功运行。

⚠️ 重要注意事项

  1. reclaimable 配置：队列的 reclaimable 字段必须显式设置为 true 才能被回收。
  2. 与 Enqueue Action 的潜在冲突：如果同时配置了 enqueue 和 reclaim，且 enqueue 因资源不足阻止 PodGroup 入队，会导致没有 Pending 状态的 Pod，进而可能无法触发 reclaim 机制。
  3. Overcommit 插件的影响：overcommit 插件可能会阻止资源超限的作业进入队列，这与回收机制可能产生冲突，在实际部署中需要留意。
  4. 层级队列：在层级队列中，资源回收可能会在兄弟队列之间进行，若兄弟队列资源不足，可能会向祖先队列查找。

💡 总结

Volcano 的 Reclaim Action 是其实现多租户公平调度和跨队列资源弹性的关键组件。它通过一套严谨的流程（识别短缺队列、选择受害者队列、模拟安全检查、执行回收）和与其他插件（尤其是 Gang 插件）的协同，确保了在回收资源时不会破坏现有作业的完整性，同时又能高效地将资源重新分配给更需要的队列。

问题一：reclaim和preempt都需要有pending状态的pod来触发，那么由于enqueue action的存在，集群资源不足时，会禁止Job创建出pod，如何处理这种矛盾

您提出的这个问题非常尖锐且完全正确，您指出了一个在配置 Volcano 时非常关键但容易被忽略的逻辑矛盾。您的推理是完全准确的：如果高优先级作业被 enqueue action 阻挡，无法创建 Pod，那么后续的 preempt action 就失去了作用对象，优先级机制也就形同虚设了。

🧠 理解冲突的本质

首先，明确一下冲突的根源：

  • Enqueue 是 “预防式” 的策略：它在作业的Pod被实际创建之前就进行资源预检。它的目标是避免在资源不足时创建大量永远无法调度的Pending Pod，从而减轻API Server的压力并避免资源碎片化。它追求的是集群稳定性。
  • Reclaim/Preempt 是 “补救式” 的策略：它们依赖于系统中已经存在Pending状态的Pod作为“触发器”。这些Pending Pod代表了未被满足的资源需求，从而驱动调度器去抢占或回收其他低优先级作业的资源。它追求的是资源利用率和调度公平性。

因此，Enqueue 和 Reclaim 在逻辑上存在一个“死锁”：

  1. 没有资源 -> Enqueue 阻止PodGroup入队 -> 没有Pending Pod被创建
  2. 没有Pending Pod -> Reclaim 没有回收资源的触发信号 -> 资源无法被释放
  3. 资源无法被释放 -> 继续没有资源 -> 回到第1步

这确实是 enqueue 和 preempt 这两个 action 在设计哲学上的一个根本性冲突。在实际生产中，为了解决这个死锁，社区形成了一种共识和最佳实践。

🧠 核心解决方案：通常选择禁用 Enqueue

正如我之前提到的，最常见的解决方案是：在 volcano-scheduler-configmap 的 actions 列表中移除 enqueue。

这样配置的工作原理和后果如下：

  1. Pod 立即创建：当一个高优先级的 Volcano Job 被提交后，volcano-controller-manager 会立即为其创建所有 Pod，无论当前集群资源是否充足。这些 Pod 会直接进入 Pending 状态。
  2. Preempt Action 获得触发器：这些 Pending 状态的高优先级 Pod 为 preempt action 提供了明确的抢占目标。preempt action 会检测到这些 Pod 因为资源不足而无法调度，但同时它们的优先级又很高。
  3. 触发抢占：preempt action 开始工作，在同一队列内寻找低优先级的、已运行的、并且抢占后不会破坏其 Gang 约束的 Pod 作为受害者。
  4. 执行抢占：驱逐（删除）受害者 Pod，释放资源。
  5. 资源再分配：在下一个调度周期，被释放的资源就可以分配给之前处于 Pending 状态的高优先级 Pod，使其得以运行。

通过这个流程，优先级的语义就得到了真正的贯彻执行。 高优先级作业通过“牺牲”低优先级作业来获得资源。

⚖️ 代价与权衡

当然，禁用 enqueue 是有代价的，这就是一个典型的权衡（Trade-off）：

  • 你获得了什么：强大的抢占能力和优先级保证。确保了高优先级工作负载能够及时获得资源，满足 SLA（服务等级协议）。
  • 你付出了什么：API Server 和 Controller 的压力。在大型集群中，可能会有成千上万个长期处于 Pending 状态的 Pod，它们会对 Kubernetes 的控制平面（特别是 API Server 和 etcd）造成额外的压力，因为所有组件都需要持续地监听和处理这些 Pod 对象。

🔧 其他辅助解决方案

除了完全禁用 enqueue，还有一些更精细化的策略来缓解这个矛盾：

  1. 按队列粒度启用/禁用 Enqueue（高级功能）：
    这是一种理想但实现起来更复杂的方式。理论上，你可以：

    • 对高优先级队列的 Job，使其绕过 enqueue 检查，确保 Pod 能被立即创建以触发抢占。
    • 对低优先级队列的 Job，仍然启用 enqueue，让它们在资源充足时才创建 Pod，避免给系统增加不必要的负担。
    • （注：Volcano 原生可能不直接支持这么精细的控制，但这可以通过开发自定义的准入控制器或修改调度器配置来间接实现）。

  2. 设置超时机制：
    即使启用了 enqueue，也可以设置一个超时时间。如果一个 PodGroup 在 Pending 状态停留了太久（例如 10 分钟），可以通过外部脚本或控制器强制将其状态改为 Inqueue，从而触发 Pod 创建和后续的抢占流程。这是一种折中方案。

💡 结论与最终建议

配置策略 优先级保障 对控制面压力 建议场景
启用 enqueue 弱。高优先级作业可能无法及时触发抢占。 小。避免了大量 Pending Pod。 资源充裕、环境稳定、对优先级要求不高的批处理场景。
禁用 enqueue 强。优先级机制能得到完美执行。 大。存在大量 Pending Pod。 主流选择。多租户、资源紧张、需要严格保证高优先级作业 SLA 的生产环境。

对于绝大多数需要优先级和抢占功能的生产环境，社区给出的明确答案是：禁用 enqueue action。

因此，您不应该在配置中同时使用 enqueue 和 preempt 而期望它们能协同工作。正确的做法是舍弃 enqueue，接受它所带来的控制面压力，以换取强大的调度能力。 这种压力通常可以通过升级控制平面硬件、优化 etcd 配置等方式来缓解。

问题二：queue中的deserved（应得资源量）、capability（资源上限）和 guarantee（保障资源量）这几个参数分别是什么作用，是如何配合的

您提出的这个问题非常核心，是理解 Volcano 多租户资源管理模型的关键。deserved、capability 和 guarantee 这三个参数共同构成了一个既公平又具弹性且安全的资源分配体系。

简单来说，它们回答了三个问题：

  1. 公平性 (Fairness)：在资源竞争时，每个队列应该分到多少？ -> deserved
  2. 隔离性 (Isolation)：每个队列最多允许使用多少？ -> capability
  3. 保障性 (Guarantee)：每个队列最少确保能得到多少？ -> guarantee

下面我们进行详细的分解。

1. 三个参数的角色与作用

参数 英文名 核心作用 类比理解
guarantee Guarantee 资源保障。队列绝对最小可使用的资源量，即使集群资源紧张，这部分资源也应由该队列独占或被回收回来。 你的基本工资。无论公司业绩多差，这笔钱必须发给你。
capability Capability 资源上限。队列绝对最大可使用的资源量，即使集群资源空闲，该队列也无法超过此限制。 你的年度薪资封顶。今年业绩再好，收入也不会超过这个数。
deserved Deserved 应得资源。队列按权重在当前集群条件下应得的资源量。这是一个动态计算的值，介于 guarantee 和 capability 之间。 你的绩效工资。根据你的贡献（权重）和公司当前利润（集群资源），今年应该发给你这么多。

2. 为什么需要这三个参数？

它们解决了多租户集群中的不同问题：

  1. **guarantee**：解决 “饿死” (Starvation) 问题。

    • 场景：如果没有 guarantee，一个权重很低但运行着重要基础服务的队列，可能会被权重高的大作业完全挤占资源，导致服务不可用。
    • 作用：它为重要队列提供了一个安全垫，确保了其业务的基线服务能力。

  2. **capability**：解决 “贪婪” (Greediness) 问题。

    • 场景：如果没有 capability，一个队列可能会无限占用所有集群资源，导致其他队列完全无法工作（例如，代码有 bug 提交了无限量的作业）。
    • 作用：它为每个队列设置了资源边界，提供了资源隔离，是多租户安全的基石。

  3. **deserved**：解决 “公平” (Fairness) 与 “弹性” (Elasticity) 问题。

    • 场景：集群资源是波动的。deserved 根据每个队列的 weight 和集群当前可用资源总量，动态计算出一个公平的分配额。
    • 作用：它使得资源分配不是僵化的，而是在保障基本盘 (guarantee) 的前提下，允许队列弹性地共享集群的空闲资源，从而最大限度地提高整体资源利用率。

3. 它们是如何协同工作的？（核心）

计算过程与交互：

  1. **计算 deserved**：

    • deserved 不是静态配置的，而是由 Proportion 插件动态计算出来的。
    • 计算公式可以简化为：queue's deserved = (queue's weight / total weight of all queues) * (available idle resources) + queue's guarantee
    • 但它最终会被钳制在 [guarantee, capability] 区间内。计算出的 deserved 值永远不能小于 guarantee，也不能大于 capability。

  2. 驱动 Reclaim Action：

    • 回收目标：Reclaim Action 会持续检查所有队列。如果一个队列的当前已使用资源 (allocated) 大于其 deserved，并且它的 reclaimable=true，那么它就成为资源回收的目标（受害者）。
    • 回收来源：同时，如果一个队列的 allocated 小于其 deserved，说明它应该得到更多资源，它就成为资源回收的发起方。
    • 回收目标值：Reclaim 的目标是将队列的资源使用量调整到其 deserved 的水平。

4. 一个具体的例子

假设一个集群有 100核 CPU。
有两个队列：

  • Queue-online:
    • weight: 2
    • guarantee: 30
    • capability: 70
  • Queue-offline:
    • weight: 1
    • guarantee: 10
    • capability: 50

场景 1：集群空闲时（两个队列都已使用各自 guarantee 的资源）

  • 空闲资源 = 100 - 30 - 10 = 60核。
  • 计算 deserved：
    • Queue-online.deserved = 30 + (2/3) * 60 = 30 + 40 = 70核 (达到capability上限)
    • Queue-offline.deserved = 10 + (1/3) * 60 = 10 + 20 = 30核 (在10-50之间)
  • 结果：Queue-online 最多可用到70核，Queue-offline 最多可用到30核。Reclaim 不会触发，因为双方都用不到自己的 deserved。

场景 2：集群繁忙时（Queue-offline 已用了50核，超过了其 deserved）

  • 假设 Queue-offline 当前用了50核（达到了它的 capability）。
  • 而 Queue-online 只用了40核（低于它的 deserved 70核）。
  • Reclaim Action 触发：
    • Queue-offline (allocated=50) > Queue-offline.deserved (~30) -> 受害者
    • Queue-online (allocated=40) < Queue-online.deserved (70) -> 发起方
  • Reclaim 执行：从 Queue-offline 中回收资源，直到它的使用量接近其 deserved 的30核。

总结

  • guarantee：是底线，确保重要队列不被“饿死”，是防御性的。
  • capability：是天花板，防止单个队列“吃撑”，是隔离性的。
  • deserved：是目标线，在底线和天花板之间动态调整，追求公平和弹性，是调度器（特别是 Reclaim Action）操作的核心依据。

这三个参数共同作用，使得 Volcano 能够实现一个真正生产级可用的、兼顾公平、效率与安全的多租户资源管理模型。

💎 总结

所以，回答你的问题：即使两个队列总和未超100核，且 queue-online 暂时无额外需求，reclaim 机制仍可能主动触发。

这并非多此一举，而是为了：

  1. 即刻的公平性：确保每个队列的资源使用量符合其 deserved 份额设定的公平原则。
  2. 规则的强制性：维护资源分配规则的权威性，确保资源配额不是“软约束”。
  3. 未来的灵活性：为其他队列可能随时出现的资源需求做好准备，保持集群资源的弹性和流动性。

可以说，reclaim 是一种积极的、预防性的调度手段，而不仅仅是被动的应急反应。它致力于在问题发生前就维持一个清晰、公平且高效的资源状态。
