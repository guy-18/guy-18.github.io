volcano-allocate-action详解

Volcano 调度器中的 Allocate Action 是调度流程的核心环节，它负责为已经入队（Inqueue）的 Pod 分配具体的节点。这个过程非常精密，涉及到预选、优选、资源预留和最终的绑定决策。

下面我将结合 Volcano 的最新源码（主要参考 v1.12.0 版本），为你详细解析 Allocate Action 的具体过程。

🔧 Allocate Action 的核心作用

Allocate Action 的核心任务是为每个已入队的 Pod 尝试分配一个合适的节点。它不仅会检查节点资源的充足性，还要确保满足作业级别的约束（如 Gang 调度策略），并在最终一次性提交所有绑定决策。

📦 关键数据结构

在深入流程之前，了解几个关键的数据结构有助于理解后续过程：

  • Session: 代表一个调度周期（Session）的上下文，包含了当前调度所需的所有信息，如待调度的 Job 列表 (Jobs)、集群节点信息 (Nodes)、已注册的插件 (Plugins) 等。Allocate Action 会操作这个 Session。
  • JobInfo: 并非直接对应 Volcano Job CRD，而是调度器内部对 PodGroup 及其所属 Pod 的封装。一个 JobInfo 包含了一个 PodGroup 下的所有 Pod（封装为 TaskInfo）。
  • TaskInfo: 调度器内部对 Pod 的封装。包含了 Pod 的详细信息以及其资源请求。
  • NodeInfo: 调度器内部对 Kubernetes Node 的封装。包含了节点的资源容量、已分配资源、支持的操作系统架构等信息。

🔍 Allocate Action 详细工作流程

Allocate Action 的入口点在 pkg/scheduler/actions/allocate/allocate.go 文件中的 Execute 函数。其工作流程可以概括为以下几个核心步骤：

  1. 准备待调度的任务：遍历当前 Session 中的所有 JobInfo，收集其中处于 Pending 状态的 TaskInfo（Pod）。这些任务就是本轮 Allocate Action 需要尝试调度的对象。
  2. 任务排序：根据配置的插件（如 Priority 插件），对这些待调度的 TaskInfo 进行排序。通常优先级高的任务会优先得到调度。
  3. 遍历任务，尝试分配：这是最核心的循环。对于每一个待调度的任务，执行以下子步骤：
    • a. 预选 (Predicate)：调用 Predicates 插件，过滤掉所有不满足该 Pod 基本要求的节点。例如，检查节点的资源（CPU、内存、GPU）是否 >= Pod 的请求，检查节点Selector、亲和性/反亲和性、污点容忍等。通过这一步，会得到一个候选节点列表。
    • b. 优选 (Prioritize/Node Ordering)：如果预选后还有多个候选节点，则会调用 NodeOrder 等插件为这些节点打分。插件会考虑多种因素，如资源平衡（Binpack 或 Spread）、节点亲和性、镜像是否已存在等，最终选出一个最优节点。
    • c. 资源预留 (Reservation)：并不是立即调用 Kubernetes API 进行绑定，而是在当前 Session 的缓存中，模拟从选出的最优节点上扣除该 Pod 所请求的资源。这一步确保了后续的调度决策是基于最新的资源视图，避免了重复分配。
    • d. 生成分配指令：将 (TaskInfo, NodeInfo) 这个分配决策记录下来，存储在 Session 中，等待后续提交。
  4. 提交分配决策 (Commit)：在所有合适的任务都经过上述流程后，Allocate Action 会一次性提交本周期内所有的分配决策。调度器会将这些绑定信息（Pod 和 Node 的对应关系）通过 Kubernetes API Server 进行更新（设置 Pod 的 nodeName 字段）。

⚙️ 依赖的关键插件

Allocate Action 本身是一个框架，其具体行为由一系列插件决定：

  • Gang 插件: 这是 “All or Nothing” 策略的强制执行者。即使一个 Pod 通过了预选和优选，Gang 插件也会检查其所属的整个 JobInfo (PodGroup) 是否满足 minAvailable 条件。只有在满足 minAvailable 时，本次会话中为该 Job 所做的所有分配才会被提交；否则，所有分配都会被回滚，该 Job 的 Pod 不会绑定到任何节点。Gang检查是在遍历完所有的pod之后。
  • Predicates 插件: 负责预选，实现节点过滤的逻辑。它检查 Pod 是否适合调度到某个节点上。
  • NodeOrder 插件: 负责优选，实现节点打分的逻辑。它为每个候选节点计算一个分数，调度器选择分数最高的节点。
  • Binpack 插件: 通常与 NodeOrder 插件协同工作，倾向于将资源分配集中在尽可能少的节点上，以提高节点利用率，便于后续腾空节点。
  • Priority 插件: 用于在多个待调度的 Task(Pod) 之间进行优先级排序，决定哪个 Pod 优先进行分配。

🧪 结合示例说明

假设我们有一个包含 1 个 ps 和 2 个 worker 的 TensorFlow 作业（PodGroup，minAvailable=3），并且已通过 Enqueue Action 检查，Pod 已被创建。

  1. Allocate Action 开始：调度器收集这 3 个处于 Pending 状态的 Pod。
  2. 任务排序：假设优先级相同，则按创建时间等顺序处理。
  3. 为 ps Pod 分配：
    • Predicate：过滤出所有满足 ps 资源请求（例如 1 CPU, 2GiB Mem）的节点。
    • NodeOrder：对候选节点打分（例如，选择资源碎片最少的节点 node-a）。
    • 预留：在 Session 中标记 node-a 的 1 CPU 和 2GiB Mem 被预留。
  4. 为 worker-1 Pod 分配：
    • Predicate：过滤节点（现在 node-a 可用资源减少了）。
    • NodeOrder：选出最优节点 node-b。
    • 预留：预留 node-b 上 2 CPU 和 4GiB Mem。
  5. 为 worker-2 Pod 分配：
    • Predicate：过滤节点。
    • NodeOrder：选出最优节点 node-c。
    • 预留：预留 node-c 上 2 CPU 和 4GiB Mem。
  6. Gang 插件检查：调度器检查到该 Job 的 3 个 Pod 都已成功分配节点，满足 minAvailable=3 的条件。
  7. 提交：调度器通过 API Server 将这 3 个 Pod 分别绑定到 node-a, node-b, node-c。

如果集群资源只够运行 2 个 worker，不够运行 ps，那么 Gang 插件会检查失败，导致所有 3 个 Pod 的分配都会被撤销，等待下一个调度周期再尝试。这就有效防止了分布式任务因部分进程无法启动而造成的死锁。

⚠️ 注意事项

  1. 执行顺序：在 volcano-scheduler-configmap 中，allocate 通常放置在 enqueue 之后，backfill 之前。Action 的执行顺序很重要。
  2. 插件配置：确保 allocate 所依赖的插件（如 predicates, nodeorder, gang）已在配置中正确启用并放置在合适的 tier 中。
  3. 与 Preempt/Reclaim 的区别：Allocate 是正常分配，而 Preempt 是同一队列内基于优先级的抢占，Reclaim 是跨队列的资源回收。它们解决的是不同层面的资源分配问题。
