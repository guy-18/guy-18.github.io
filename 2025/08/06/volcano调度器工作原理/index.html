volcano调度器工作原理

Volcano 调度器核心概念与工作原理详解

一、核心概念全景图

1. 调度会话 (Session)

  • 定义：调度器每次决策的完整周期
  • 生命周期：
    graph LR
  A[会话开始] --> B[获取集群快照]
  B --> C[执行调度逻辑]
  C --> D[提交调度决策]
  D --> E[会话结束]
  • 关键数据：包含集群状态（节点、作业、队列）的完整内存模型

2. 调度单元 (Scheduling Units)

概念 描述 K8s 原生对应
PodGroup 原子调度单元（一组必须整体调度的Pod） 无
VolcanoJob 增强型Job（支持多任务模板） Job
Queue 资源队列（多租户隔离） Namespace ResourceQuota
Task 作业内部的任务单元 PodTemplate

3. 调度策略插件 (Plugins)

插件名 功能 关键参数
Gang 保证作业完整性 minAvailable
DRF 多资源公平调度 weight
Binpack 高密度装箱 binpack.resources
Priority 作业优先级 priorityClassName
Topology 拓扑感知调度 topologyPolicy
Preempt 抢占机制 preemptable

4. 调度动作 (Actions)

动作 执行顺序 功能
Enqueue 1 作业入队（检查队列配额）
Allocate 2 核心资源分配
Backfill 3 碎片资源填充
Preempt 4 抢占低优先级作业
Reclaim 5 回收超额资源

三、核心调度场景解析

1. Gang Scheduling 实现

问题：如何保证分布式作业的所有Pod同时启动？

解决方案：

sequenceDiagram
    participant S as Scheduler
    participant C as Cluster
    S->>C: 检查minAvailable资源
    alt 资源充足
        S->>C: 预留所有资源
        S->>C: 原子绑定所有Pod
        C-->>S: 绑定成功
    else 资源不足
        S->>C: 不绑定任何Pod
        C-->>S: 保持Pending状态
    end

2. 抢占式调度 (Preemption)

场景：高优先级作业需要资源时

工作流程：

graph TD
    A[新作业提交] --> B{资源不足}
    B -->|是| C[选择牺牲者]
    C --> D[驱逐低优先级Pod]
    D --> E[释放资源]
    E --> F[调度新作业]

牺牲者选择策略：

  1. 最低优先级作业
  2. 最小进度损失（可恢复作业优先）
  3. 最少Pod数量

3. 拓扑感知调度 (Topology Awareness)

  3. 确保GPU通过高速互联

四、调度器高级特性

1. 批调度优化

策略 描述 适用场景
Binpack 紧密装箱减少碎片 资源紧张环境
Spread 分散部署提高可用性 高可用需求
DRF 多资源公平分配 多租户环境

2. 弹性资源管理

graph LR
    A[资源监控] --> B{资源不足}
    B -->|是| C[驱逐离线作业]
    B -->|否| D[恢复离线作业]
    D --> E[从检查点继续]

3. 任务依赖 (DAG)

五、调度器内部状态机

PodGroup 状态流转

stateDiagram-v2
    [*] --> Pending 创建
    Pending --> Running 资源满足
    Running --> Completed 所有任务成功
    Running --> Failed 任务失败
    Pending --> Failed 超时/配置错误
    Failed --> Pending 重试（maxRetry）

调度决策状态

状态 描述 常见原因
Unschedulable 无法调度 资源不足/约束冲突
Wait 等待依赖 DAG前置任务未完成
Pending 排队中 队列配额不足
Scheduled 已调度 成功分配节点
