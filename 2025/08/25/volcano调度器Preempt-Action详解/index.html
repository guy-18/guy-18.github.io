volcano调度器Preempt-Action详解

通过对 Volcano 源码（主要是 v1.12.0 版本）的分析，我来为你详细解析 Preempt Action 的具体过程。Preempt Action（抢占动作）是 Volcano 调度器中用于处理同一队列内高优先级任务抢占低优先级任务资源的核心机制。

🔧 Preempt Action 的核心作用与设计初衷

Preempt Action 的核心目标是当集群资源不足时，确保高优先级的 Job 或 Task 能够优先获得资源，及时完成计算任务。它通过选择性驱逐低优先级的 Pod 来释放资源，满足高优先级任务的调度需求。

值得注意的是，Volcano 的 Preempt Action 通常用于同一队列（Queue）内的抢占。这与 Reclaim Action 不同，后者主要负责跨队列的资源回收。

📦 关键数据结构与前置知识

在深入了解流程之前，明确几个关键概念和数据结构：

  • Session: 调度会话的上下文，包含了当前所有待调度的 Job(Jobs)、节点信息(Nodes)、已注册的插件(Plugins)等。
  • JobInfo: 调度器内部对 PodGroup 的封装，包含了优先级信息（通过 priortyClassName 获取）。
  • TaskInfo: 调度器内部对 Pod 的封装，也拥有优先级信息。
  • 优先级来源: 优先级通常来源于 Pod 或 Job 的 priorityClassName 字段，其值映射到一个整数 Value，值越大，优先级越高。

🔍 Preempt Action 详细工作流程

Preempt Action 的入口点在 pkg/scheduler/actions/preempt/preempt.go 文件中的 Execute 函数。其工作流程可以概括为以下核心步骤：

  1. 识别受害者（Victim Selection）：
    Preempt Action 会遍历所有当前已调度（处于 Running 或 Bound 状态）的 TaskInfo（Pod），寻找那些优先级低于当前待调度任务的 Pod。这些低优先级的 Pod 被标记为“潜在的”抢占受害者。

  2. 模拟抢占与安全检查：
    对于每一个待调度的高优先级任务和每一个潜在的受害者，调度器会进行模拟抢占。这个过程严重依赖 Gang 插件：

    • Gang 插件会检查：如果驱逐（抢占）了某个（或某些）低优先级的 Pod，是否会导致其所属的 Job 不满足 minAvailable 约束（即破坏其 Gang 调度策略）。
    • 同时，Conformance 插件会确保 kube-system 命名空间下的关键 Pod 不会被抢占。
    • 只有那些被驱逐后不会破坏其原 Job 完整性的低优先级 Pod，才会被确定为“真正的”受害者。

  3. 执行抢占：
    一旦确定了可行的受害者，调度器会真正地驱逐这些 Pod。这是通过调用 Kubernetes API 删除这些 Pod 来实现的。这些被抢占的 Pod 会进入 Terminating 状态，其资源将被释放。

  4. 资源释放与后续调度：
    受害者 Pod 被删除后，它们之前占用的资源（CPU、内存等）也就释放了。这些释放出来的资源不会立即分配给发起抢占的高优先级任务。而是等待下一个调度周期，由 Enqueue 和 Allocate 等 Action 来重新分配这些资源。此时，由于高优先级任务仍在队列中，并且资源变得可用，它就有很大概率在后续的调度中被成功分配资源。

⚙️ 依赖的关键插件

Preempt Action 的具体行为由一系列插件决定：

  • Gang 插件: 这是安全检查的核心。它确保了抢占行为不会破坏分布式任务的原子性（”All or Nothing”）。这是 Preempt Action 能安全工作的基石。
  • Priority 插件: 用于比较 JobInfo 或 TaskInfo 之间的优先级，确定谁可以抢占谁。
  • Conformance 插件: 提供系统层面的安全保护，确保 kube-system 命名空间下的关键 Pod 不会被抢占，维护集群的稳定性。

⚠️ 重要配置与注意事项

要使 Preempt Action 正常工作，必须在 volcano-scheduler-configmap 中进行正确配置：

关键参数解析：

  • enablePreemption: 必须设置为 true 来全局启用抢占功能。
  • enableJobStarving: 这个参数的名称有些误导性。将其设置为 false 通常意味着允许低优先级任务被抢占（即“挨饿”），从而为高优先级任务让路。设置为 true 反而可能会保护它们不被抢占。

🧪 结合示例说明

假设队列 queue-a 中有两个 Job：

  • Job-Low: 优先级为 10，已运行 5 个 Pod（其 minAvailable 为 3）。
  • Job-High: 优先级为 100，需要 3 个 Pod，但因资源不足处于 Pending。

当 Preempt Action 执行时：

  1. 识别受害者: 发现 Job-Low 的优先级 (10) 低于 Job-High (100)，其 Pod 被列为潜在受害者。
  2. 模拟与检查: Gang 插件检查抢占 Job-Low 的 2 个 Pod 是否可行。由于 Job-Low 的 minAvailable 是 3，抢占 2 个后还剩 3 个运行中的 Pod，满足条件，安全检查通过。如果 minAvailable 是 5，则抢占任何 1 个 Pod 都会导致其不满足条件，抢占会被禁止。
  3. 执行抢占: 驱逐 Job-Low 的 2 个 Pod。
  4. 资源释放: 这 2 个 Pod 占用的资源被释放。
  5. 后续调度: 在下一个调度周期，Job-High 的 3 个 Pod 因为优先级高，在 Allocate action 中成功分配到这部分新释放的资源。

💡 总结与关键点

  1. 队列内抢占: Volcano 的 Preempt Action 主要处理同一队列内的抢占。
  2. 安全第一: 抢占过程受到 Gang 策略的严格约束，绝不会以破坏一个分布式应用的完整性为代价去满足另一个应用。
  3. 异步过程: 抢占（删除 Pod）和资源的再分配（调度高优先级 Pod）是跨调度周期的异步过程。
  4. 配置是关键: 必须正确配置调度器 ConfigMap 中的 actions 和 preemption 插件参数（特别是 enablePreemption: true）才能启用抢占功能。
  5. 系统保护: 关键系统 Pod 默认受到保护，不会被抢占，确保了集群的稳定性。
